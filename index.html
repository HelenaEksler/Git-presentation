Git: Basics

What is it?

By far, the most widely used modern version control system in the world today is Git. Git is a mature, actively maintained open source project originally developed in 2005 by Linus Torvalds


When you do actions in Git, nearly all of them only add data to the Git database. It is hard to get the system to do anything that is not undoable or to make it erase data in any way. As in any VCS, you can lose or mess up changes you haven’t committed yet; but after you commit a snapshot into Git, it is very difficult to lose, especially if you regularly push your database to another repository.

File statuses


Git has three main states that your files can reside in:

  • Untracked
  • Unmodified
  • Modified
  • Staged
  • Commited

Files status changes

      To check files status
      $ git status
      To move file from Untracked or Modifide to Staged:
      $ git add <filename>
      To add all files:
      $ git add *
      To see not staged changes:
      $ git diff
      To see staged changes:
      $ git diff --staged
      To commit files:
      $ git commit <filename>
      To commit all staged files:
      $ git commit
      To commit all modified, staged and untracked files:
      $ git commit -a
      To commit with specific message:
      $ git commit -m"<Your message>"

Ignoring files

You can also ignore some files in order not to track them and their changes.
For this - simply add file (or folder) to .gitignore file:


/config/config-local.php
/uploads
/uploads/*

Note: If you are trying to ignore file that is already tracked - it won't be ignored even if it was added to .gitignore file.

Undoing things

On any step of the work you can undo things. Git provides couple of ways of doing this:


      Unstage changes:
      $ git reset HEAD <filename>
      Remove untracked files:
      $ git clean
      Revert all currently not commited changes:
      $ git reset --hard
      Revert not staged changes:
      $ git checkout -- <filename>
      Revert commit:
      $ git revert <commit>
      Revert the commit we just created:
      $ git revert HEAD

Remotes

Remote branches

To be able to collaborate on any Git project, you need to know how to manage your remote repositories. Remote repositories are versions of your project that are hosted on the Internet or network somewhere. You can have several of them, each of which generally is either read-only or read/write for you. Collaborating with others involves managing these remote repositories and pushing and pulling data to and from them when you need to share work.

      Command to add new remote:
      $ git remote add <shortname> <url>
      Command to see the list of remotes:
      $ git remote

Forks

So when you are forking repository - you are creating a related copy of the project, and each fork can be added as a remote.

Branches

Nearly every VCS has some form of branching support. Branching means you diverge from the main line of development and continue to do work without messing with that main line.
Also you could use the branches when you work on different tasks at same time and you want to keep these changes separately.


To move to existing branch use command:
$ git checkout <branchname>

To move to new branch use command:
$ git checkout -b <branchname>

Note: The branch a new one will be compared to is the one you have done checkout from.


And for each branch you can create Merge request.
All changes from the branch can be "Merged" into the master branch.

Keeping versions up-to-date

Keep your fork and local version up to date with the main. For this we should from time to time take the recent changes from the main remote to our local master branch, push them into the fork master, and also it's part of the best practices to re-roll all the branches from the recent master.


      To have the master branch up to date with the main remote:
      $ git fetch <remote> - This will bring all changes to your local version of the remote.
      $ git checkout master
      $ git merge <remote>/master - This will put changes from remote master into your local master.
      $ git push origin master - This will put changes from your local master into your fork.

      Instead of fetch and merge you can use one command - pull.
      $ git checkout master
      $ git pull <remote> master
      $ git push origin master
      To re-roll the feature branches from master:
      $ git checkout <branch>
      $ git merge <remote>/master

Stash

If you have some changes locally and want to move for a second to other branch for other task, but your work is not finished yet - you propaply don't want to commit not finished work, but definately want to save changes. For this you can use "stash" system, which will take you local not-commited changes, and put them to the side. After you finish with other stuff - you can get back to your branch and bring your changes back from stash.

      Put your changes to stash:
      $ git stash
      To see which stashes you’ve stored:
      $ git stash list
      $ stash@{0}: WIP on master: 049d078 added the index file
      $ stash@{1}: WIP on master: c264051 Revert "added file_size"
      $ stash@{2}: WIP on master: 21d80a5 added number to log
      To apply the last stash:
      $ git stash apply
      To apply the last stash:
      $ git stash apply <stash id>

The workflow

      1. Fork the project:
      2. Clone the fork repository and add the main as the other remote:
      $ git clone https://gitlab.com/helenaeksler/test.git
      $ git remote add main https://gitlab.com/ewave/test.git
      3. Start working in the separate branch:
      $ git checkout -b bugfix
      $ git status
      $ git add *
      $ git commit -m"My commit with the bug fix"
      4. Push your changes to the fork repository:
      $ git push origin bugfix
      And create the Merge Request - compare the branch bugfix and main/master
      5.Evey day update your local and fork repositories:
      $ git checkout master
      $ git pull main/master
      $ git push origin master
      $ git checkout bugfix
      $ git merge origin/master
